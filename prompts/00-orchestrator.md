# 00 — Оркестратор: порядок и зависимости

Этот файл — для **тебя (человека-оркестратора)**, не для агента. Описывает, в каком порядке
раздавать задачи и что от каждой ждать.

## Граф зависимостей

```
01-env-setup
      │
02-data-pipeline
      │
03-train-2b   (отладка пайплайна)
      │
04-train-8b   (целевое обучение)
      │
05-eval       (оценка + прогон эталонов)
      │
06-report     (SOLUTION.md + упаковка + публикация)
```

Линейно. Каждая следующая задача стартует, только когда предыдущая помечена ✅ в `STATE.md`.

## Как запускать каждую задачу

1. Открой `STATE.md`, проверь, что предыдущая задача ✅ и артефакты на месте.
2. Скопируй содержимое `prompts/NN-*.md` целиком в новую агент-сессию.
3. Дождись, пока агент допишет `STATE.md` и положит артефакты в `solution/`.
4. Бегло проверь критерий приёмки из промпта. Если не выполнен — верни агенту с замечанием.

## Параллелизм
Почти всё линейно. Единственная возможная параллель: пока идёт долгое `04-train-8b`,
можно заранее готовить `05-eval` (скрипты оценки) и `06-report` (шаблон SOLUTION.md) —
но обучать и оценивать по факту последовательно (одна GPU).

## Ручные шаги (не для агентов)
- Этап 9: вставить ссылку на решение в форму платформы, пройти анкетирование.
- Этап 10: учесть рецензию, согласовать оценку в вузе.

## Если агент застрял
- Блокер по железу/стеку → см. `CONTEXT.md §4` (footgun'ы Blackwell/WSL2).
- Противоречие в данных → прав `CONTEXT.md`, агент должен сообщить, а не угадывать.
- Пометить задачу ⛔ blocked в `STATE.md` с причиной.
