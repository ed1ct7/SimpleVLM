# Лог технических решений (ADR-lite)

Короткие записи: решение + почему. Агенты дописывают сюда, когда делают значимый выбор.

## D1. Рецепт LLaVA (а не end-to-end)
**Решение:** строить VLM по рецепту LLaVA (энкодер + проектор + LLM, 2-стадийное обучение).
**Почему:** проверенный стандарт, дешевле end-to-end (по статье VLMs Explained KOSMOS-2-подход
дороже), deepvk-эталоны построены так же → честное сравнение.

## D2. QLoRA вместо полного файнтюна
**Решение:** обучать через QLoRA (4-bit базовая модель + LoRA-адаптеры).
**Почему:** VRAM всего 16 ГБ. Полный файнтюн 8B не влезает; QLoRA 8B ~11–14 ГБ — помещается
с grad checkpointing и micro-batch.

## D3. Старт с 2B, цель 8B
**Решение:** сначала отладить пайплайн на модели класса 2B (Gemma-2b/Qwen2-2B), потом
целевое обучение Saiga-8b.
**Почему:** быстрый цикл отладки данных/кода без долгих прогонов; 8B — для максимальной
метрики и сравнения с llava-saiga-8b.

## D4. WSL2, не нативный Windows
**Решение:** весь ML-стек в WSL2 (Ubuntu).
**Почему:** bitsandbytes/flash-attn/deepspeed на нативном Windows нестабильны; в WSL2 всё
ставится как на Linux, RTX 5080 виден через CUDA.

## D5. Свежий стек под Blackwell
**Решение:** CUDA 12.8+, PyTorch 2.7+ (cu128), свежие bitsandbytes/triton.
**Почему:** RTX 5080 — Blackwell (sm_120). Старые сборки падают `no kernel image available`.

## D6. Энкодер для отладки — CLIP ViT-L/14-336 (SigLIP — кандидат на метрику)
**Решение (этап 03):** отладочный цикл собран на **`openai/clip-vit-large-patch14-336`** —
канонический энкодер рецепта LLaVA, нативно 336px (как требует CONTEXT), без трения в
`transformers.LlavaForConditionalGeneration` (стратегия `default`, слой −2 → 576 image-токенов).
**Почему именно сейчас:** цель этапа — рабочий цикл, не метрика; CLIP-336 даёт предсказуемое
число токенов и минимум совместимостных рисков. **SigLIP не отвергнут:** `build_vlm`
авто-детектит тип энкодера (siglip → стратегия `full`, слой −1) — переключение = одна строка в
YAML (`encoder_id: google/siglip-so400m-patch14-384`). **Окончательный выбор CLIP vs SigLIP —
на этапе 05 (eval), по метрике** на GQA-ru/MMBench-ru.
**Подводный камень (зафиксирован):** процессор LlavaProcessor при `strategy="default"` отдаёт
`num_patches−1=575` image-токенов, а энкодер (после дропа CLS) — 576 фич → merge падает
(«Image features and image tokens do not match»). Лечится `num_additional_image_tokens=1`
(см. `solution/train/model.py:build_processor`).

## D7. (открыто) Инструмент оценки: lmms-eval vs свой скрипт

## D9. База 8B = `IlyaGusev/saiga_llama3_8b`; meta-init сборки (этап 04)
**Решение:** целевая LLM — **`IlyaGusev/saiga_llama3_8b`** (Llama-3-8B, рус., НЕ gated). Это та же
база, что у эталона **`deepvk/llava-saiga-8b`** (проверено: его `text_config._name_or_path` =
`IlyaGusev/saiga_llama3_8b`, vision = `clip_vision_model` 336px/patch14/layer −2/`default`).
То есть наш `build_vlm(encoder=clip-vit-large-patch14-336, llm=saiga_llama3_8b)` повторяет
архитектуру эталона → честное сравнение на этапе 05.
**Почему не `deepvk/llava-saiga-8b` напрямую:** это уже собранная VLM (эталон для сравнения), а
нам нужна **базовая LLM** для своего 2-стадийного обучения с нуля проектора.
**meta-init (`build_vlm(meta_init=True)`):** скелет `LlavaForConditionalGeneration(config)`
создаётся на meta-устройстве (`accelerate.init_empty_weights`), реальные модули пересаживаются,
проектор материализуется и инициализируется (normal 0.02). Снимает транзиентные ~32 ГБ RAM от
случайного 8B-тела на CPU (STATE 03, заметка #3). Проверено: дым 8B собирается, пик VRAM 12 ГБ.

## D10. Стадия 1 (alignment) на подвыборке LLaVA-Instruct-ru (нет рус. caption-корпуса)
**Решение:** канонический LLaVA stage-1 учит проектор на коротких image-caption парах
(LCS-558K / CC3M). Русского аналога в коллекции VK нет (этап 02 скачал только инструкции).
Поэтому стадия 1 (только проектор, энкодер+LLM заморожены) идёт на **подвыборке самого
`LLaVA-Instruct-ru`** как warmup проектора, лосс по всей последовательности (`mask_prompt=false`).
**Почему ок:** цель стадии 1 — выровнять визуальные фичи с пространством эмбеддингов LLM, а не
выучить инструкции; для этого годятся и инструкционные пары. На практике проектор сходится
быстро (token-acc ~0.03→0.64 к ~25 шагу). Стадия 2 (QLoRA, assistant-only loss) уже на инструкциях
по рецепту. **Память (RTX 5080 16 ГБ):** 4-bit Saiga + bf16-проектор + grad-checkpointing +
batch=1 × grad_accum=16, картинки 336px (576 image-токенов), `max_length=768`. Пик VRAM ~12 ГБ —
CPU-offload оптимизатора (был в запасе) не понадобился.

## D8. Сопоставление COCO по id, а не по имени файла (этап 02)
**Решение:** картинки `LLaVA-Instruct-ru` резолвить по 12-значному id, а не по точному имени.
Резолвер понимает 2014-нейминг (`COCO_train2014_<id>.jpg`) и 2017 (`<id>.jpg`); качаем
`train2014` по умолчанию.
**Почему:** проверено на данных — поле `image` = `coco/train2017/<id>.jpg` (2017-нейминг),
хотя CONTEXT/`docs/datasets.md` указывали 2014. Это те же снимки (LLaVA построен на train2014),
поэтому `train2014` (~13 ГБ) покрывает все id. Сопоставление по id делает пайплайн устойчивым
к набору COCO (2014 или 2017) и к расхождению нейминга. Также: реальный `train`-split =
109 905 записей (доки говорили 144k). CONTEXT/datasets.md как источник плана не правились —
расхождение задокументировано здесь, в `STATE.md` и `solution/data/README.md`.
