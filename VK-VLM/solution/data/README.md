# Пайплайн данных VK-VLM (этап 02)

Готовит набор для SFT: тексты `deepvk/LLaVA-Instruct-ru` + сопоставленные изображения COCO,
плюс манифест `(изображение, диалог)` в chat-формате целевой LLM.

Все данные кладутся в **WSL-ФС** (`~/vk-vlm-data`, ext4), **не** на `/mnt/*` — медленный I/O
тормозит обучение (CONTEXT §4).

---

## ⚠️ Важный нюанс про изображения (проверено на данных)

CONTEXT/`docs/datasets.md` говорят про **COCO 2014**. По факту поле `image` в датасете =
`coco/train2017/<id>.jpg` (**2017-нейминг**, имя файла `000000253464.jpg`). Это **те же
снимки**, что лежат в COCO **train2014** (LLaVA-Instruct целиком построен на `train2014`).

Поэтому пайплайн сопоставляет картинку **по 12-значному id**, а не по точному имени файла:

| Поле в датасете | Физический файл (train2014) |
|---|---|
| `coco/train2017/000000253464.jpg` | `coco/train2014/COCO_train2014_000000253464.jpg` |

Резолвер (`_common.build_coco_index`) понимает оба нейминга, так что качать можно `train2014`
(по умолчанию, ~13 ГБ, покрывает **все** id) **или** `train2017`/`val2017` — не важно.

Ещё одно расхождение с доками: `train`-split содержит **109 905** записей (не 144k).

## Файлы

| Файл | Назначение |
|---|---|
| `download.py` | тянет датасет (`save_to_disk`) + качает/распаковывает COCO в `~/vk-vlm-data/coco/` |
| `build_dataset.py` | резолвит картинки по id, отбрасывает без файла (с логом), пишет манифест JSONL + `stats.json` |
| `sample.py` | 5 случайных примеров: печать диалога + проверка, что картинка открывается в PIL (размеры > 0) |
| `_common.py` | общее: корень данных, индекс COCO по id, конвертация в chat-формат |

## Запуск (в WSL2, активированном venv из этапа 01)

```bash
source ~/vk-vlm-env/bin/activate
cd /mnt/d/Fork/SimpleVLM/VK-VLM        # репозиторий (код на /mnt — ок; данные — нет)

# 1) загрузка: текстовый датасет + COCO train2014 (~13 ГБ, докачиваемо)
python solution/data/download.py
#   при желании добить val2014: --coco-splits train2014,val2014

# 2) сборка манифеста (картинка ↔ диалог), chat-формат
python solution/data/build_dataset.py
#   --template text --image-token "<image>"   # плоский формат вместо структурного
#   --limit 100                                # быстрый прогон

# 3) проверка: 5 примеров + открытие картинок
python solution/data/sample.py
```

## Chat-формат (выбор шаблона — параметр `--template`)

Роли: `human → user`, `gpt → assistant`, `system → system`. Изображение крепится один раз
(к первой реплике с токеном `<image>`).

- **`messages`** (по умолчанию) — структурный `content`, рекомендуется для TRL/processor;
  конкретный chat-template модели применяется уже на обучении (этап 03):
  ```json
  {"role": "user", "content": [{"type": "image"},
                               {"type": "text", "text": "Что на картинке?"}]}
  ```
- **`text`** — плоский `content`-строка, плейсхолдер задаётся `--image-token`:
  ```json
  {"role": "user", "content": "<image>\nЧто на картинке?"}
  ```

## Статистика

`build_dataset.py` печатает и пишет в `processed/stats.json`:
**всего записей / с валидной картинкой / отброшено** (+ примеры отброшенных `image`).

Фактический полный прогон (`processed/stats.json`, COCO train2014 распакован, 82 783 jpg):

| Метрика | Значение |
|---|---|
| Всего записей (`train`) | **109 905** |
| С валидной картинкой | **109 905** |
| Отброшено (нет файла) | **0** |
| Картинок в индексе COCO | 82 783 |
| Манифест | `processed/llava_instruct_ru.jsonl` (109 905 строк, 196 МБ) |

> **0 отброшенных** подтверждает: все id из `coco/train2017/...`-путей лежат в COCO **train2014**
> (резолвер по id сшивает 2017-нейминг ↔ 2014-файл). `sample.py` открыл **5/5** картинок
> (напр. 500×376, 640×428 JPEG). Доп. `val2014` не нужен.
>
> Примечание про «144k»: у датасета есть и split `validation` (34 075). 109 905 (train) +
> 34 075 (val) = 143 980 ≈ 144k — отсюда цифра в доках. Для SFT используем `train`.
