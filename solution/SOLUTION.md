# SOLUTION — обучение русскоязычной VLM на открытых данных VK

> Подробное описание решения (обязательное требование VK №4). Источник правды по фактам —
> [`CONTEXT.md`](../CONTEXT.md); журнал прогона и метрики — [`STATE.md`](../STATE.md);
> технические решения — [`docs/decisions.md`](../docs/decisions.md).

## 1. Постановка задачи и цель (требование №1)

**Цель.** Обучить рабочую русскоязычную визуально-языковую модель (VLM) на **открытых данных
VK** (коллекция `deepvk` на Hugging Face) по рецепту LLaVA, воспроизвести архитектуру эталона
`deepvk/llava-saiga-8b` и **честно измерить** её на двух бенчмарках — **`GQA-ru`** (открытые
вопросы, accuracy) и **`MMBench-ru`** (выбор A/B/C/D, 20 навыков, accuracy) — единым
воспроизводимым протоколом, сравнив с эталонами deepvk; разрыв с эталоном проанализировать.

**Задачи.** (1) поднять окружение под Blackwell/16 ГБ; (2) собрать данные (LLaVA-Instruct-ru +
COCO); (3) отладить пайплайн на 2B; (4) обучить целевую 8B (2 стадии + QLoRA); (5) оценить на
обоих бенчмарках единым протоколом; (6) упаковать решение.

> «Максимальная метрика» / «обойти эталон» в брифе VK — **пример** цели, не порог сдачи (бриф:
> «может звучать так»). Обязательные требования VK — 4 (см. §8). Обойти эталон вынесено в
> **stretch-цель** (§7): главный рычаг — обучение на полном манифесте, за рамками базовой сдачи.

**Что такое VLM.** На вход — изображение + текст, на выход — текст. Архитектура из трёх частей:
**визуальный энкодер → проектор (MLP) → языковая модель (LLM)**. Рецепт обучения — **LLaVA**.

**Ожидаемый результат.** Рабочий, воспроизводимый пайплайн (данные → обучение → оценка) и
обученная целевая VLM класса 8B, сопоставимая по архитектуре с эталоном `deepvk/llava-saiga-8b`,
с честно измеренными метриками на обоих бенчмарках.

**Достигнутый результат (кратко).** Цель выполнена: пайплайн собран и воспроизводим end-to-end;
обучены две модели (отладочная 2B и целевая 8B) по 2-стадийному рецепту LLaVA + QLoRA на 16 ГБ
VRAM; целевая 8B связно отвечает по-русски по картинке; обе эталонные модели deepvk и обе свои
прогнаны через **единый честный протокол** оценки на обоих бенчмарках. Метрики свои **ниже
эталонов** — ожидаемо: свои модели обучены на **подвыборках** (debug-grade объём, лимит
времени/VRAM), эталоны — на полном корпусе. Разрыв проанализирован (§6), путь его сокращения
(обойти эталон = stretch) описан (§7). Базовые требования сдачи (§8) — закрыты.

## 2. Открытые данные VK — что и КАК использовано (требование №2)

Все данные — из коллекции **deepvk «Vision-Language Modeling»** на Hugging Face.

| Ресурс | Тип | Роль в решении | Как именно использован |
|---|---|---|---|
| `deepvk/LLaVA-Instruct-ru` | датасет, 109 905 train | **обучение (SFT)** | Источник инструкционных диалогов «вопрос по картинке → ответ». Стадия 1 (alignment) — подвыборка 6 000; стадия 2 (instruction tuning) — подвыборка 15 000. |
| **COCO 2014** (`train2014`) | изображения, 82 783 jpg | **обучение (картинки)** | В `LLaVA-Instruct-ru` **нет файлов картинок** — только привязка по id. Картинки берутся из открытого COCO 2014 и сшиваются с датасетом. |
| `deepvk/GQA-ru` | бенчмарк, 80.1k | **оценка** | `testdev_balanced`: join `*_instructions` ⋈ `*_images` по `imageId`; сабсет 1000 (seed 7), accuracy exact + lenient. |
| `deepvk/MMBench-ru` | бенчмарк, 3.91k | **оценка** | Весь `dev` (3910), single-choice A/B/C/D, accuracy по распарсенной букве. |
| `deepvk/llava-saiga-8b` | модель 8B | **эталон** | Прогнан через тот же протокол оценки — верхняя планка для своей 8B. |
| `deepvk/llava-gemma-2b-lora` | модель 3B | **эталон** | Прогнан через тот же протокол — планка для своей 2B. |

### Критичный нюанс про изображения (как именно сшиваются)

Поле `image` в `LLaVA-Instruct-ru` = `coco/train2017/<id>.jpg` (**2017-нейминг**, напр.
`000000253464.jpg`), хотя это **те же** снимки, что в **COCO train2014** (LLaVA построен на
train2014). Поэтому сопоставление сделано **по 12-значному COCO-id**, а не по имени файла
(резолвер `solution/data/_common.py:build_coco_index` понимает 2014- и 2017-нейминг). Это делает
пайплайн устойчивым к набору COCO. Результат: при сборке манифеста **0 записей отброшено** —
все 109 905 путей сшились с COCO train2014 (`solution/data/processed/stats.json`). Подробности —
[`docs/decisions.md` D8](../docs/decisions.md).

> Расхождение с документацией зафиксировано (не скрыто): доки обещали `train2014`-нейминг и
> 144k записей; реально — 2017-нейминг и 109 905 train (+ 34 075 val = 144k суммарно).

## 3. Архитектура и метод обучения

### Компоненты (целевая 8B повторяет эталон `deepvk/llava-saiga-8b`)

```
изображение 336×336
      │
      ▼
CLIP ViT-L/14-336  (openai/clip-vit-large-patch14-336)
   слой −2, дроп CLS → 576 image-токенов
      │
      ▼
проектор: 2-слойный MLP (LlavaMultiModalProjector)   ← обучается целиком
      │  визуальные фичи → пространство эмбеддингов LLM
      ▼
LLM: Saiga-8b (IlyaGusev/saiga_llama3_8b, Llama-3-8B рус.)   ← 4-bit nf4, QLoRA-адаптеры
      │
      ▼
текстовый ответ (рус.)
```

Сборка — через `transformers.LlavaForConditionalGeneration`: `LlavaConfig` из двух под-конфигов
+ пересадка реальных весов энкодера/LLM (`solution/train/model.py:build_vlm`), проектор
инициализируется заново. Выбор Saiga обоснован тем, что это **та же база, что у эталона**
(проверено по его config: `text_config._name_or_path = IlyaGusev/saiga_llama3_8b`), и она **не
gated** → честное архитектурное сравнение ([D9](../docs/decisions.md)).

### Рецепт LLaVA — 2 стадии

1. **Стадия 1 (alignment).** Энкодер и LLM **заморожены**, обучается **только проектор** —
   научить его мэппить визуальные фичи CLIP в пространство эмбеддингов Saiga. Лосс по всей
   последовательности (`mask_prompt=false`). Артефакт — `solution/model/8b-stage1/projector.safetensors`.
2. **Стадия 2 (instruction tuning).** Проектор стартует из стадии 1; совместное дообучение
   **проектора + LLM через QLoRA**, **assistant-only loss** (учим отвечать, не повторять промпт).

> Канонический stage-1 LLaVA учит проектор на коротких caption-парах (LCS-558K/CC3M). Русского
> аналога в коллекции VK нет, поэтому стадия 1 идёт на подвыборке самого `LLaVA-Instruct-ru`
> как warmup проектора — для выравнивания фич этого достаточно ([D10](../docs/decisions.md)).

### QLoRA и экономия памяти (VRAM-стена 16 ГБ)

- База 8B — **4-bit nf4 + double-quant** (bitsandbytes), compute bf16.
- **LoRA r16/α32** на `q,k,v,o,gate,up,down`; проектор — `modules_to_save` (обучается +
  сохраняется в адаптерах). Энкодер заморожен, **в bf16** (квантование энкодера ломает image-фичи).
- **gradient checkpointing + micro-batch=1 + grad_accum + paged_adamw_8bit + `max_length=768`**
  (576 image-токенов + текст; жёсткий потолок активаций) + `expandable_segments`.
- **meta-init** сборки (`accelerate.init_empty_weights`) — без транзиентных ~32 ГБ RAM от
  случайного 8B-тела на CPU.

## 5. Оценка — единый протокол

Своим скриптом (`solution/eval/`), **не** `lmms-eval`: у последнего нет готовых тасков под
HF-форматы `deepvk/GQA-ru` / `MMBench-ru`, а свой скрипт даёт **один протокол на все 4 модели** +
прямой контроль формата вывода MMBench ([D7](../docs/decisions.md)).

- **Greedy**-декодирование; каждая модель рендерится **своим процессором** (свой image-токен +
  chat-шаблон своей LLM). База 4-bit, **vision_tower/projector/lm_head — bf16** (квантование
  ломает image-фичи — проверено на эталоне).
- **GQA-ru**: `testdev_balanced`, N=1000 (seed 7). `exact` — строгое совпадение нормализованного
  ответа (стандарт GQA, штрафует многословие); `lenient` — gold целым словом в ответе (честнее
  между моделями разной краткости).
- **MMBench-ru**: весь `dev`, N=3910; промпт требует букву A/B/C/D, парсер берёт первую.
  `letter_rate=100%` у **всех** моделей — формат вывода корректен (критерий приёмки этапа).

### Нетривиальные находки оценки (диагностированы и пофикшены)

- **Gemma-нормализатор:** `llava-gemma-2b-lora` под transformers 5.12 на картинках генерил мусор.
  Корень — Gemma домножает эмбеддинги на `sqrt(hidden)`, а image-фичи идут мимо → ~×45 меньше
  текстовых. Фикс — forward-hook на проектор `×sqrt(hidden)` (гейт по `model_type==gemma*`).
  Числа в таблице — с фиксом. Saiga (Llama-3) нормализатора не имеет.
- **Эталоны 4-bit:** `llm_int8_skip_modules` для vision_tower/projector/lm_head — иначе image-фичи
  в мусор. **BOS по-модельно** + мульти-eos стоп-токены.

## 6. Таблица метрик и анализ

Accuracy (%). Цифры совпадают с [`STATE.md`](../STATE.md) и
[`solution/results/metrics.md`](results/metrics.md). Сырые предсказания — `results/raw/*.jsonl`.

| Модель | GQA-ru extracted (exact) | MMBench-ru (acc) |
|---|---|---|
| `deepvk/llava-gemma-2b-lora` (эталон 3B) | 45.50 (45.40) | 62.99 |
| `deepvk/llava-saiga-8b` (эталон 8B) | 54.30 (54.00) | 70.15 |
| **Моя 2B** (Qwen2-1.5B, отладка) | 21.60 (1.40) | 34.63 |
| **Моя 8B** (Saiga-8b, целевая) | 28.60 (6.60) | 57.08 |

GQA-ячейка — **extracted (exact)**: `extracted` извлекает короткий ответ из (возможно
многословного) предсказания и сверяет с gold, одинаково для всех моделей ([D11](../docs/decisions.md));
`exact` (в скобках) — строгий стандарт GQA, штрафующий многословие.

### Анализ

- **Свои модели ниже эталонов своего класса на обоих бенчмарках.** Главная причина — **объём
  обучения**: эталоны обучены на полном корпусе с несколькими эпохами, свои — на **подвыборках**
  (6k/15k из 109 905, 1 эпоха) как debug-grade прогон под лимит времени/VRAM. Это **ожидаемо** и
  задокументировано. Цель проекта (обучить + честно измерить, CONTEXT §6) — **выполнена**;
  обойти эталон — stretch-цель (§7), её рычаг — полный объём обучения.
- **Многословность своих моделей маскировала качество на GQA.** Строгий `exact` у 8B = 6.6, но
  `extracted` (короткий ответ из ответа) = **28.6** — правильный ответ часто присутствовал, но был
  завёрнут в лишний текст. Эталоны краткие → `extracted≈exact` (54.0→54.3), т.е. метрика их не
  завышает: разрыв реальный (недообучение), но почти вчетверо меньше, чем казалось по голому exact.
- **MMBench менее чувствителен** к многословию (ответ — одна буква): здесь своя 8B (57.08)
  заметно ближе к эталону (70.15) и **выше своей 2B** (34.63) — видно, что 2-стадийный рецепт +
  assistant-only loss + бóльшая LLM реально работают.
- **8B > 2B по всем метрикам** — масштаб LLM и полный 2-стадийный рецепт (против single-stage
  отладки 2B) дают системный прирост.

### Хранение и анализ результатов — БД (SQLite)

Результаты оценки сложены в реляционную БД **SQLite** (`solution/db/`, схема в 3НФ): справочники
`model`/`benchmark`, факт-таблица метрик `eval_run` (звезда) и таблицы предсказаний
`gqa_prediction`/`mmbench_prediction` (4000 + 15640 строк). Загрузчик `build_db.py` наполняет её
из сырых артефактов прогонов; аналитика — SQL-запросами (`queries.sql`): лидерборд через `JOIN`,
разрез точности по навыкам через `GROUP BY category`, оценка эффекта метрики `extracted`
(`correct=0 AND correct_extracted=1`), разбор ошибок и разрыва с эталоном (self-join по `idx`).
Детали — [`db/README.md`](db/README.md).

## 7. Выводы — что повлияло и что улучшать дальше

**Что подняло метрику:**
- 2-стадийный рецепт LLaVA (отдельный alignment проектора) + **assistant-only loss** на стадии 2.
- Бóльшая и русскоязычная LLM (Saiga-8b) вместо отладочной 2B.
- Архитектурный паритет с эталоном (тот же CLIP-336 + та же база Saiga) → сравнение честное.

**Что улучшать дальше (базовая цель §1 выполнена; ниже — путь к stretch-цели «обойти эталон»,
приоритет по ожидаемому эффекту):**
1. **Полный объём обучения** — `subset 109905` + несколько эпох на обеих стадиях (команда и
   оценка времени — `solution/train/RUN-8b.md`). Главный рычаг к паритету с эталоном.
2. **Краткость ответа** — `extracted` (D11) уже снял формат-штраф на eval (6.6→28.6); следующий
   шаг — учить краткости на обучении (терсе-ответы в данных/промпте), чтобы и строгий `exact` рос.
3. **Энкодер SigLIP** (`google/siglip-so400m-patch14-384`) — кандидат на прирост; переключение =
   одна строка в YAML, `build_vlm` авто-детектит тип ([D6](../docs/decisions.md)).
4. **Бюджет адаптеров** — выше LoRA r/α, больше target-модулей; при наличии VRAM — длиннее
   `max_length`.
