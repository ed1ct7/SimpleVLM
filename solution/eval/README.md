# Оценка (этап 05) — GQA-ru + MMBench-ru, протокол и воспроизведение

Измеряем accuracy своей VLM и эталонов deepvk на двух бенчмарках по **единому протоколу**
(CONTEXT §6 — честное сравнение). Инструмент — **собственный скрипт** (решение D7 в
`docs/decisions.md`): у `lmms-eval` нет готовых тасков для `deepvk/GQA-ru` и `deepvk/MMBench-ru`
(это HF-датасеты deepvk, не каноничные GQA/MMBench), а MMBench требует жёсткого контроля
формата вывода (буква A/B/C/D) — свой скрипт даёт и то, и другое + один прогон на все 4 модели.

## Модели (реестр `eval_common.MODELS`)

| Ключ | Что | Как грузится |
|---|---|---|
| `mine-8b` | Моя 8B (`solution/model/8b`) | `build_vlm`(CLIP+Saiga) + PEFT-адаптеры (как `train/infer.py`) |
| `mine-2b` | Моя 2B (`solution/model/2b-debug`) | `build_vlm`(CLIP+Qwen2-1.5B) + PEFT-адаптеры |
| `ref-saiga-8b` | `deepvk/llava-saiga-8b` | `LlavaForConditionalGeneration.from_pretrained` |
| `ref-gemma-2b` | `deepvk/llava-gemma-2b-lora` | `LlavaForConditionalGeneration.from_pretrained` (готовая merged-VLM, не LoRA) |

Все четыре — LLaVA-архитектуры (CLIP ViT-L/14-336 → MLP-проектор → LLM). База в **4-bit**
(nf4 + double-quant, compute bf16); визуальная башня и проектор — в **bf16** (квантование
энкодера ломает image-фичи). У эталонов процессор шипнут с `patch_size=None` → проставляем
из `vision_config` (иначе `LlavaProcessor` падает на подсчёте image-токенов); chat-шаблон
берём из токенайзера, если в процессоре его нет.

**BOS — по-модельно** (`VLM.answer`): если chat-шаблон уже вставил BOS (Llama-3/Saiga) →
`add_special_tokens=False` (иначе двойной BOS); если не вставил (Gemma) → `True` (без BOS
Gemma выдаёт мусор). Стоп-токены — все из `<|eot_id|>/<|im_end|>/<end_of_turn>/eos` (иначе
утечка "assistant"-заголовка в хвост ответа).

**Gemma-фикс (важно):** `deepvk/llava-gemma-2b-lora` под transformers 5.12 на картинках выдаёт
мусор (китайский/тагальский), хотя веса грузятся и текст-онли работает («Париж»). Причина:
Gemma масштабирует входные эмбеддинги на `sqrt(hidden_size)` (нормализатор), а LLaVA вставляет
image-фичи в `inputs_embeds` сырыми, мимо нормализатора → они ~×45 меньше текстовых → модель
их игнорирует. Llama-3/Saiga нормализатора не имеет — не страдает. Лечится хуком на проектор:
`out *= sqrt(hidden_size)` (гейт по `text_config.model_type == gemma*`). После фикса gemma даёт
связные ответы по картинке (метрики в `metrics.md` — с фиксом).

## Единый протокол

Едины для всех моделей: набор/сабсет, текст задачи, параметры генерации (**greedy**,
`do_sample=False`), метрика и парсинг. Различается только рендер — **каждая модель использует
свой процессор**: свой `image_token` и свой chat-шаблон LLM (Llama-3 / Gemma / Qwen2).
Навязывать чужой шаблон было бы нечестно (модель обучалась под свой).

Промпт: image-токен в начале (`{image_token}\n{текст}`), затем `apply_chat_template(...,
add_generation_prompt=True)` — как на обучении (`data_collator.flatten_messages`).

### GQA-ru (открытые вопросы → accuracy exact-match)
- Источник: `deepvk/GQA-ru`, конфиги `testdev_balanced_instructions` (12 216 вопросов) +
  `testdev_balanced_images` (398 картинок), join по `imageId`. Это каноничный
  **testdev-balanced** сабсет GQA.
- **Сабсет:** по умолчанию **1000** вопросов, фиксированный `seed=7` (один и тот же сабсет для
  всех 4 моделей — выборка зависит только от порядка датасета и сидов, не от модели). Полный
  набор — `--n 0` (12 216). Сабсет взят ради времени прогона на 16 ГБ; протокол это допускает.
- Инструкция: «Ответь на вопрос кратко — одним словом или короткой фразой.»
- Метрика (`eval_score.py`): headline **`extracted`** — короткий ответ, извлечённый из (возможно
  многословного) предсказания, сверенный с gold (да/нет → первое слово-полярность; иначе gold как
  ведущее слово/среди первых трёх; объединён с exact → не ниже строгого). Одинаково для всех
  моделей (D11). Также: **`exact`** — строгое равенство нормализованных (`norm_answer`: lower, ё→е,
  выкинуть пунктуацию, схлопнуть пробелы; стандарт GQA, штрафует многословие); **`lenient`** —
  диагностический потолок (gold целым словом где угодно в ответе).
- Пересчёт метрики из готового raw без GPU: `python solution/eval/rescore_gqa.py` (полезно после
  смены логики парсинга — модели по картинкам не перегоняются).

### MMBench-ru (single-choice A/B/C/D → accuracy)
- Источник: `deepvk/MMBench-ru`, split `dev` — **3 910** вопросов (единственный split с ответами).
- **Полный dev** по умолчанию (`--n 0`).
- Промпт перечисляет варианты (`A. … B. … …`, опции `nan` отбрасываются — есть вопросы на 2–3
  варианта) + `hint` (если есть) и **требует**: «Ответь только буквой варианта (A, B, C или D).»
- Парсинг: первая буква `A–D` в ответе (`parse_letter`). `letter_rate` в meta фиксирует долю
  ответов, где буква реально найдена — **проверка критерия приёмки** (модель выдаёт букву, а не
  свободный текст; иначе метрика ложно занижена).
- **Circular eval** (ротация вариантов) НЕ делаем — одиночный проход (быстрее, метрика
  сопоставима между всеми моделями на одном протоколе).

## Запуск

Из корня репо, venv `~/vk-vlm-env`, в WSL2. Полный прогон (4 модели × 2 бенчмарка) **одним**
фоновым процессом (изоляция по процессам → чистая VRAM между моделями):

```bash
wsl -d Ubuntu-24.04 -- bash -lc \
  "source ~/vk-vlm-env/bin/activate && bash solution/eval/run_eval.sh \
     > solution/eval/logs/run-eval.log 2>&1"
```

Параметры через env: `GQA_N` (деф. 1000, 0=весь), `MMB_N` (деф. 0=весь dev), `SEED` (деф. 7).

Отдельная модель/бенчмарк:

```bash
python solution/eval/eval_gqa.py     --model mine-8b      --n 1000
python solution/eval/eval_mmbench.py --model ref-saiga-8b --n 0
python solution/eval/aggregate.py    # свести raw/*.meta.json → results/metrics.md
```

## Артефакты

- `eval_common.py` — общая библиотека (загрузка моделей/бенчмарков, промпты, метрики, прогон).
- `eval_gqa.py`, `eval_mmbench.py` — CLI на одну модель/бенчмарк.
- `run_eval.sh` — полный прогон + агрегация. `aggregate.py` — сборка таблицы.
- `solution/results/metrics.md` — итоговая таблица модель × бенчмарк × accuracy.
- `solution/results/raw/<bench>__<model>.jsonl` — сырые предсказания (по строке на вопрос:
  gold, pred, correct). `<bench>__<model>.meta.json` — сводка прогона (accuracy, n, тайминги,
  `letter_rate`).

## Заметки

- Память (RTX 5080 16 ГБ): одна модель за раз; `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`.
- `HF_HOME=~/vk-vlm-data/hf-cache` (WSL-ФС). Веса эталонов докачиваются при первом обращении
  (saiga-8b ~16 ГБ, gemma ~5 ГБ); датасеты берутся из кэша.
- Детерминизм: greedy + фиксированный сабсет (seed) → прогоны воспроизводимы.
- **Резюмируемость:** `run_benchmark` дописывает сырой jsonl и пишет `.meta.json` только когда
  обработаны ВСЕ вопросы. Если прогон убит (WSL реапит отвязанные процессы между вызовами
  `wsl --`, STATE 04 #1 — **не запускать другие `wsl --` команды, пока идёт фоновый прогон**),
  повторный запуск той же команды докатывает остаток (готовые id пропускаются). Завершённые
  (модель × бенчмарк) при повторе пропускаются мгновенно.
- **GQA extracted vs exact (D11):** мои модели многословны (STATE 03 #5) → строгий exact их сильно
  занижает (8B exact 6.6); `extracted` извлекает короткий ответ → 28.6, честно. Эталоны отвечают
  кратко → их extracted≈exact (54.0→54.3), т.е. метрика их не завышает. В таблице headline
  `extracted (exact)`, `lenient` — потолок в доп. таблице.
